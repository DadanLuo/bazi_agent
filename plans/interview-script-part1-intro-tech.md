## 一、项目自我介绍（90秒版本）

"好的，我来介绍一下这个项目。

这个项目叫做**赛博司命**，是一个基于 LangGraph 的智能八字命理分析 Agent 系统，目前版本是 0.2.0。

从用户视角看，它能做什么？用户输入出生日期、时辰、性别、出生地，系统会自动完成八字排盘，输出五行分析、格局判定、喜用神推断、流年运势、大运走向，最终由大模型生成一份个性化的命理解读报告，并支持多轮追问。

从技术视角看，核心挑战有三个：
第一，传统命理计算的精度问题，我们实现了基于天文算法的真太阳时校正，精确到经纬度；
第二，古籍知识的检索准确性问题，我们实现了 Vector + BM25 的混合检索配合 Rerank 重排序；
第三，对话体验问题，我们实现了意图路由 + 槽位填充 + 会话记忆的完整多轮对话链路。

整体架构上，前端是单页面应用，后端是 FastAPI + LangGraph，存储是 Redis + 文件双写，大模型是通义千问 qwen-plus。

这个项目的特别之处在于：它不是简单地把提示词扔给 LLM 让它'算命'，而是构建了一套真实的规则引擎做底层计算，再由 LLM 做语言层面的解读，所以准确性和可解释性都比较有保障。"

## 二、为什么做这个项目（动机阐述）

"这个项目有几个出发点：

第一，**技术选型的完整性**。我想在一个真实业务场景下，把 LangGraph 工作流、RAG 检索增强、Agent 抽象层这些当前 AI 应用开发的主流技术都系统地用一遍，而不是拼凑 demo。

第二，**领域的复杂度**。八字命理是一个规则密集型的领域，有完整的五行生克体系、格局判定规则、喜用神推导逻辑，这非常适合用来验证 LangGraph 多节点状态机的设计能力。

第三，**工程性考验**。这个场景对高可用有真实要求：调用大模型可能超时、Rerank API 有调用次数限制、Redis 可能不可用，每种情况都需要对应的降级处理，不能让系统崩溃。

所以这个项目不只是一个 demo，它覆盖了 API 设计、工作流编排、RAG 实现、缓存策略、限流超时、会话管理这些生产级关注点。"

## 三、技术选型详解

### 3.1 为什么用 FastAPI？

"后端框架选择 FastAPI，主要有三个理由：

**第一，原生异步支持**。FastAPI 基于 Starlette，天然支持 async/await。我们的核心路径——调用大模型、调用 Rerank API——都是 I/O 密集型操作，用异步可以在单进程内并发处理多个请求，不需要多线程的复杂性。具体来说，整个 LangGraph 工作流通过 `bazi_app.ainvoke()` 异步执行，LLM 调用通过 `llm.acall()` 异步执行，不会阻塞事件循环。

**第二，类型安全**。FastAPI 基于 Pydantic v2 做请求响应校验，比 Flask 的手工校验省力很多，而且错误响应格式统一，便于前端处理。我们的 `ChatInput`、`ChatResponse`、`BirthInfo` 都是 Pydantic 模型，自动生成 OpenAPI 文档，也方便调试。

**第三，中间件生态**。Starlette 的中间件模型是洋葱模型，我们在这上面实现了限流、超时、CORS、GZip、结构化日志五层中间件，注册顺序就决定了执行顺序，非常清晰。"

---

### 3.2 为什么用 LangGraph？

"工作流编排选择 LangGraph，而不是直接写顺序代码，原因是：

**第一，状态可观测**。LangGraph 用 TypedDict 定义全局状态 `BaziAgentState`，每个节点读取状态、写入状态，整个流程在任意时刻都可以 dump 出完整的 JSON，便于调试和问题定位。如果我用 Python 函数链式调用，中间状态就散落在各个局部变量里，出问题很难排查。

**第二，条件路由**。我们有 11 个节点，每个节点执行完都需要判断是继续向下还是跳到 `safety_check` 兜底。LangGraph 的 `add_conditional_edges` 让这种分支逻辑变得声明式，不需要到处写 if-else。

**第三，可扩展性**。如果后续要加节点，比如加一个'大运详细分析'节点，只需要 `workflow.add_node()` 加进去，在合适的位置加边，不需要改其他节点的代码。

**相比 LangChain Chain**，LangGraph 是图结构，可以有循环、有并发节点（虽然我们目前没用）；LangChain Chain 是线性的 pipe，不适合这种多分支的工作流。"

---

### 3.3 为什么用 ChromaDB + DashScope Embedding？

"向量存储选择 ChromaDB，理由是：

**本地部署、零运维**。ChromaDB 支持 persistent 模式，数据存在本地 `src/chroma_db/` 目录，不需要额外的向量数据库服务。对于我们这个规模的知识库（古籍文本，几千条 chunk），ChromaDB 完全够用。

**HNSW 索引**。在 v0.2.0 中我们做了 HNSW 索引优化，向量检索速度相比默认模式提升了约 30%。HNSW 是一种基于图的近似最近邻算法，通过多层跳跃连接把检索复杂度从 O(n) 降到 O(log n)。

**Embedding 选择 DashScope text-embedding-v4**，是因为它对中文文本的语义理解效果好，尤其是古文词汇，比通用的多语言 Embedding 更准确。而且我们的 LLM 也用 DashScope，统一用一个 API Key，运维更简单。"

---

### 3.4 为什么用 BM25 + 向量混合检索，而不是单纯向量检索？

"这是一个关键的工程决策。

纯向量检索的问题是：**对专有名词不敏感**。比如用户问'正印格喜用神'，如果知识库里有一段'正印格，月令正印，日主得印生扶'，向量相似度可能不够高，因为'正印格'这个词的语义在 Embedding 空间里不一定跟查询向量很近。

BM25 是关键词频率检索，它对**精确字符串匹配**特别强。'正印格'在 BM25 里权重很高，能精准召回包含这个词的段落。

但 BM25 的问题是**不理解语义**。比如'日主偏弱需要印比帮扶'，如果用户问'八字身弱怎么补救'，BM25 可能检索不到，因为关键词不匹配。

所以**混合检索**取两者之长：向量检索负责语义理解，BM25 负责关键词召回，融合分数公式是：

```
fused_score = 0.6 * vector_normalized_score + 0.4 * bm25_normalized_score
```

向量权重 0.6、BM25 权重 0.4，这个比例是经过调优的，古籍文本中关键词命中的重要性略低于语义相似性。

最后再加一层 **DashScope gte-rerank** 重排序，把融合后的 top-20 候选文档重新精排，选出最相关的 5 篇，进一步提升准确性。这三层检索形成了 Recall → Rerank 的二阶段架构。"

---

### 3.5 为什么选通义千问 qwen-plus？

"大模型选通义千问 qwen-plus，有几个考量：

**第一，中文能力强**。古籍解读、命理术语的中文表达，qwen-plus 比很多通用模型更地道。

**第二，16K context window**。我们的完整八字分析 prompt 包括用户数据、五行分析结果、古籍检索内容，加起来可能有几千 token，16K 的窗口足够容纳，不需要截断。

**第三，DashScope 一站式**。Embedding、Rerank、LLM 统一用 DashScope，只需要一个 API Key，账单统一，不需要管理多个服务的认证。

当然，这个架构设计上做了抽象。我们有 `BaseLLM` 抽象类，`DashScopeLLM` 是它的一个实现。如果后续要换成 OpenAI 或者本地部署的 Qwen，只需要实现一个新的子类，不需要改调用方代码。"

---

### 3.6 为什么用 Redis + 文件双写而不是纯数据库？

"存储层的设计决策：

**Redis 的作用是热数据缓存**。两个场景：第一，相同出生信息的八字排盘结果（TTL 2 小时），避免重复走完整个 LangGraph 流程；第二，会话消息（TTL 24 小时），支持多设备访问同一个对话。

Redis 的问题是**非持久化**，重启或内存满了数据就没了。所以我们同时维护**文件存储**作为持久化层，每次会话保存都是 Redis + 文件双写。

**文件存储的格式**是 gzip 压缩的 JSON，保存在 `data/memory/{conversation_id}.json`。之所以不用关系型数据库，是因为会话数据是非结构化的，每个会话的消息数量、八字缓存字段都不固定，JSON 比关系型表更灵活。

**降级策略**：如果 Redis 连接失败，系统不会报错，只走文件存储，对用户无感知。这是通过 `try/except` 包装 Redis 操作实现的，Redis 失败时直接 `return None`，业务逻辑走 cache miss 的分支。"

## 四、技术选型总结（面试结尾升华版）

"总结一下技术选型的核心原则：

1. **选合适的，不选最新的**。ChromaDB 够用，不需要 Pinecone；文件存储够用，不需要 MongoDB。
2. **每个选型都要能说出 trade-off**。FastAPI vs Flask、LangGraph vs LangChain Chain、混合检索 vs 单一检索，每个都有取舍，不是随便选的。
3. **可降级可替换**。所有外部依赖——Redis、LLM、Rerank——都做了降级处理，系统的核心功能（八字计算）在任何外部服务不可用时都能正常运行。
4. **抽象层保护业务逻辑**。`BaseAgent`、`BaseLLM`、`BaseRetriever` 这些抽象类让具体实现可以替换，业务逻辑层不依赖具体的第三方 SDK。"

## 五、常见追问与应答

**Q: 这个项目生产上线了吗？**

"目前是内测版本，通过 Docker 容器化部署在云服务器上，支持并发访问。没有做水平扩展，但架构上已经为水平扩展做了准备——会话状态存在 Redis 而不是进程内存，所以理论上多个实例共享同一个 Redis 就能水平扩展。"

**Q: 项目团队规模？**

"个人项目，我独立设计并实现，历时约三周完成 v0.1.0，后续两周迭代到 v0.2.0，加入了 Redis 缓存、HNSW 索引、限流超时等生产级特性。"

**Q: 最大的技术难点是什么？**

"有两个：第一是八字月柱的计算精度。月柱不是按公历月份算，而是按节气换月，我们实现了基于天文算法的节气精确计算，考虑了夏令时和真太阳时校正；第二是 LangGraph 状态管理和会话系统的对接，LangGraph 的状态是每次请求独立的 TypedDict，而会话系统需要跨请求持久化，这两套数据模型的转换设计花了不少时间，最终通过 `UnifiedSession.to_graph_state()` 和 `absorb_graph_result()` 两个方法解决了互相转换的问题。"

**Q: 下一步计划？**

"v0.3.0 计划引入流式输出（Streaming Response），用户不需要等 LLM 生成完才看到内容；另外考虑接入微信小程序端，以及加入更多命理模型比如紫微斗数的支持。"
